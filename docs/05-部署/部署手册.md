## 7. 常见问题

| 现象 | 原因 | 解决 |
|------|------|------|
| 本机通过 Tailscale 访问 e540 很慢 | 本机为对称 NAT，导致与 e540 直连失败，长期经 `DERP(hkg)` 中继 | 优先通过 222 做 SSH 跳板访问 e540，必要时通过 222 做端口转发 |

### 7.1 Tailscale / e540 / 222 连通性排查结论

#### 问题背景

在远程部署 / 运维过程中，本机通过 Tailscale 访问 e540 体感明显变慢，影响 SSH 登录与后续操作稳定性。为确认瓶颈位置，针对本机、e540、222 三者之间的网络路径进行了连通性排查。

#### 排查项

- e540 的 NAT 类型是否异常
- 本机 → e540 的 Tailscale 路径是否长期经过官方 DERP
- 本机 → 222 的公网延迟是否足够低
- 222 → e540 的 Tailscale / SSH 路径是否可用
- 222 是否具备自建 derper 的基础条件

#### 排查结果

- e540 NAT 正常（`MappingVariesByDestIP: false`），不是当前慢路径的主要问题源
- 本机 `tailscale netcheck` 显示 `MappingVariesByDestIP: true`，说明本机侧更接近对称 NAT，直连能力较差
- 本机 `tailscale ping 100.64.247.48` 实测长期经 `DERP(hkg)`，RTT 约 `430ms ~ 1.3s`，且出现 `direct connection not established`
- 本机公网访问 222（`101.71.198.93`）平均延迟约 `30.4ms`
- 222 已加入同一 tailnet（节点 `svc-222`），并确认具备以下条件：
  - Go / OpenSSL / systemd 可用
  - `3478/udp` 空闲
  - `80/443` 已被 nginx 占用，`4433/tcp` 空闲
- 从 222 到 e540 的 `tailscale ping` 可建立直连，实测可降到约 `28ms`
- 已实测 `ssh -J 81.70.24.222 e540` 可成功登录 e540

#### 结论

当前“本机访问 e540 很慢”的根因不在 e540，而在于：

1. 本机侧 NAT 条件较差，导致无法与 e540 建立稳定直连
2. 当前协商路径长期落在官方 `DERP(hkg)`，带来 `430ms+` 的高延迟中继访问
3. 222 与 e540 之间可以建立低延迟直连，因此 222 适合作为远程访问 e540 的跳板机

#### 推荐访问方式

- 在不改造官方 Tailscale 控制面的前提下，优先使用 222 作为 SSH 跳板：

```bash
ssh -J 81.70.24.222 e540
```

- 如需访问 e540 上的 Web / 服务端口，可通过 222 做本地端口转发，例如：

```bash
ssh -L 8080:100.64.247.48:8080 81.70.24.222
# 然后在本机访问 http://127.0.0.1:8080
```

#### 补充说明

- 222 具备部署自建 derper 的基础条件，但当前 tailnet 使用的是官方 Tailscale SaaS 控制面，自定义 DERP 的接入方式受限
- 因此，本阶段更推荐优先采用“222 跳板 + 端口转发”的低改造方案，而不是先推进自建 DERP 接入
