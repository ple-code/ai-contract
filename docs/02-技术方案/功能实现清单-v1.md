# 明衡 · AI 合同审阅系统 — 功能实现清单 v1

> 本文档基于 `app/api`（FastAPI 后端）与 `app/web`（React 前端）的实际代码逐模块盘点，反映 **当前已实现** 的能力，作为功能交付基线。
> 更新日期：2026-06-29。

---

## 0. 系统概览

| 维度 | 实现 |
|------|------|
| 产品定位 | 面向企业办公网内网的「AI 合同审阅工作台」，MVP 聚焦 **采购合同** |
| 后端 | Python 3.11 + FastAPI + SQLAlchemy 2.x（asyncpg）+ PostgreSQL 16 |
| 前端 | React 19 + TypeScript + Vite 8 + React Router 7（纯 SPA，Nginx 托管） |
| AI 接入 | OpenAI / Anthropic 双协议兼容网关；token Fernet 静态加密入库 |
| 部署 | `docker-compose.yml`（postgres / api / web）+ `start.sh` / `dev.sh` |
| 鉴权 | JWT（HttpOnly Cookie，7 天）+ bcrypt；账号分配制 + RBAC（管理员/普通用户）|
| 首次登录 | 内置管理员 `admin / Admin@123`（`seed_admin` 注入）|

## 2. 合同上传与解析

| 功能 | 实现要点 | 代码位置 |
|------|----------|----------|
| 文件上传 | 支持 `.docx` 与 `.pdf`；`FormData` 上传；UUID 命名存到 `storage/uploads/` | `routers/contracts.py::create_contract` |
| 解析器注册 | `parser_registry` 注册 docx / pdf 两个解析器，按扩展名分发 | `services/parser_registry.py` |
| DOCX 解析 | 按条款编号正则（第X条 / 一、 / 1、 / （1））切分；**表格转 markdown** 保留结构；正确处理合并单元格（gridSpan/vMerge） | `services/doc_parser.py` |
| PDF 解析 | `pdfplumber` 抽文字版；扫描件检测（>半数页文本<20字）→ 走 OCR | `services/pdf_parser.py` |
| OCR | PaddleOCR + OpenCV（去红印章 inpaint）；按行坐标排序输出 | `services/ocr_service.py` |
| 合同类型识别 | 关键词加权打分（采购合同关键词字典）；得分≥12 判定 `purchase`，否则 `unknown` 拒绝入库 | `services/type_detector.py` |
| 合同编号抽取 | 三套正则（合同编号 / 合同号 / Contract No.）；模板占位返回 None | `doc_parser.extract_contract_no` |
| 合同摘要 | 上传时一次性抽取 甲方/乙方/签订时间地点/条款结构，存 `version.summary` | `doc_parser.extract_contract_summary` |
| 条款标签 | 12 类 type_tags（价格/付款/违约金/交付/质保/保密/知识产权/合同效力/解除终止/争议解决/不可抗力/数据安全），按关键词命中打标 | `doc_parser._detect_tags` |

## 3. 重复合同识别

上传"新合同"时两级防撞（`mode=new` 且未 `confirm_duplicate`）：

| 层级 | 实现 | 代码位置 |
|------|------|----------|
| 1️⃣ 编号确定性匹配 | 解析出的合同编号与库内 `Contract.no` 字符串相等即撞号 | `routers/contracts.py` |
| 2️⃣ AI 语义相似度 | 编号没撞时，把新合同摘要 + 同类型最近 20 份库内合同的摘要喂给 LLM，返回 `{match_id, score, reason}`；阈值 ≥70 视为同一合同 | `services/similarity_checker.py` |
| 性能保障 | 比对只读 `version.summary`，不重新解析条款；甲方为空（模板）与已填甲方（真合同）强制判不相似 | `similarity_checker._gather_candidates` |
| 用户处置 | 命中后前端弹框三选一：① 关联为新版本 ② 仍作为新合同（带 `confirm_duplicate=true` 跳过检查）③ 取消 | `UploadPage.tsx` |

## 4. 版本管理与比对 Diff

| 功能 | 实现要点 | 代码位置 |
|------|----------|----------|
| 多版本归档 | 一个合同多个 `ContractVersion`；当前版本指针 `current_version_id` | `models/contract.py` |

| 关键字段变更摘要 | 正则抽取 合同总价/付款账期/违约金比例/交货期 的字段级 from→to | `services/field_extractor.py` |
| 字段格式化 | 金额去逗号转 float 再三位分隔；天/日单位归一 | `field_extractor._fmt_money` |

## 5. AI 初审（核心）

| 功能 | 实现要点 | 代码位置 |
|------|----------|----------|
| 三立场初审 | 一次调用同时生成 **甲方 / 乙方 / 中立** 三个立场的结果（每条款 × 3）；切换立场零延迟（直接查库） | `services/review_engine.py` |
| 流式进度 | SSE（Server-Sent Events）按条款 finding 实时推送；进度按 `条款数 × 3` 累计 | `routers/reviews.py` + `WorkbenchPage` |
| 关键字段抽取 | 初审开头先调一次 LLM 抽 total_price/payment_terms/penalty_rate/delivery_date/warranty_period | `review_engine` + `FIELD_EXTRACT_PROMPT` |
| 法律条文注入 | 按条款 type_tags 用 PG 数组重叠查询命中的 `LegalArticle`，拼进 prompt 限定「只能引用这些条文，不得杜撰」 | `services/legal_service.py` |
| 确定性规则注入 | 按条款命中关键词的 `ReviewRule`（如「账期>60天」「违约金>30%」）拼进 prompt，要求 AI 命中即点名 | `services/rule_service.py` |
| 立场化输出 | 每条 finding 含 risk_level / finding / suggestion / legal_basis / stance_note | `prompts/review_clause.py` |
| JSON 容错解析 | 去 markdown 围栏 + 抽首个 `{...}` 段，兼容 LLM 不严格输出 | `review_engine._parse_json` |
| 立场锁定 | 一旦有任何复审留痕（决定/批注/应用），禁止重新初审 | `review_engine.is_stance_locked` |

## 附录：当前已知限制

1. **合同类型**：仅支持采购合同（`type_detector` 关键词字典只覆盖采购类）。
2. **PDF**：仅支持文字版 PDF 与扫描件 OCR；OCR 依赖 PaddleOCR，模型较大。
3. **知识库**：法条/规则只读，不支持前端录入或上传。
4. **数据留存/水印/字段级加密**：UI 展示为默认基线，MVP 阶段不可配置。
5. **passlib 兼容**：临时 pin `bcrypt<4.1`，待迁移到直接使用 bcrypt。
