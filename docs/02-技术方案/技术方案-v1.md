### 1.2 核心数据流（一次完整审阅）

0. **登录**（F16）→ 选择**岗位**（销售/法务/商务/财务，可"记住"）→ 进入**合同列表/首页**（F18）：查看已上传合同，或点击「上传合同」（可选新合同/已有合同新版本）。立场（甲/乙/中立）在工作台内切换（F4）。
1. 法务上传合同（Word/PDF）→ **文档解析**抽取为结构化「条款树」→ **合同类型识别**（F17）：识别为采购合同则继续（界面标注类型）、入合同列表（F18），否则提示"暂不支持"并列出支持类型 → 进入工作台即可**直接定位/跳转到条款段落**（F14）。
2. 选择**比对基准**（我方标准合同文件 或 该合同上一版本）→ **比对引擎**对齐条款，产出变更集（新增/删除/修改 + 行内 diff）。
3. 按当前立场角色 → **初审编排**：对每条（重点是变更条款 + 关键字段）调用模型网关，结合**法律条文库（静态注入）** + **规则库**，产出风险点、修改建议、法律依据、立场化解读。
4. 前端渲染：风险摘要 + 条款双栏对照 + AI 解读条；法务**接受/拒绝/批注** → 写入版本并**留痕**。
5. 全过程模型调用经**开源模型网关**（默认本地 DeepSeek 分组），敏感合同不出内网。

## 二、技术选型

| 层 | 选型 | 理由 | 备选 |
|---|---|---|---|
| 前端框架 | **React 18 + TypeScript + Vite** | 团队通用、生态成熟、SPA 交互（双栏 diff、流式渲染）友好；TS 利于条款/变更结构建模 | Vue 3（同样可行，按团队栈定） |
| UI 实现 | **设计 Token 化 CSS（按 UI 基准）+ 轻量组件**；按需引 Radix/Headless UI | 原型是高保真自定义风格（藏青+金+衬线），直接 Token 化还原成本低于套用重型组件库 | Ant Design（快但风格需大改） |
| 后端 | **Python 3.11 + FastAPI** | AI/RAG/文档解析生态在 Python 最全；FastAPI 异步 + SSE 流式输出契合初审场景 | Node(NestJS)（前后端同栈，但 AI 库偏弱） |
| 文档解析 | **python-docx**（docx）+ **PyMuPDF/pdfplumber**（pdf） | docx 能读段落/表格/修订；PyMuPDF 取文本+坐标，利于"定位到原文" | Apache Tika（统一但偏重、定位弱） |
| 比对/diff | **自研条款对齐 + difflib/RapidFuzz + 字符级 diff** | 合同比对需先"条款对齐"再做行内 diff，纯文本 diff 不够；轻量可控 | python-redlines、git diff（粒度/语义不够） |
| 数据库 | **PostgreSQL 16** | 关系型强一致，合同/版本/条款/变更/留痕天然关系数据；事务+审计可靠 | MySQL（可，pgvector 生态稍弱） |
| 法条检索 | **不做向量 RAG（MVP）**：法条静态导入 PG，按"条款类型/关键词标签"筛选后直接注入 prompt | 法律数据量很小（采购相关法条至多百级），向量检索收益低、还要多维护 embedding + 向量库；直接注入更简单可控、零额外中间件 | pgvector / Qdrant（**仅当法条库显著变大（多合同类型扩充）时再上**，设计已预留 legal_article 表） |
| 文件存储 | **MinIO**（或本地卷） | 内网自托管、S3 兼容，存原始合同与各版本文件 | 直接本地文件系统（MVP 够用） |
| 模型推理 | **本地 DeepSeek（vLLM 或 Ollama，OpenAI 兼容端口）** | 数据自主可控硬约束；OpenAI 兼容便于统一网关 | DeepSeek 官方/GLM/GPT/中转站（仅测试/非敏感，见 F15） |
| 模型网关 | **LiteLLM Proxy**（单容器，YAML 配置）；或 **应用内适配层**（零额外服务） | 我们只需"路由到本地 DeepSeek + 少数供应商 + 自定义 Key"，LiteLLM 专为此而生、轻量、OpenAI 兼容、自带 fallback/限流/统一日志，"只配不改码"（F15）；若供应商极少，应用内适配层更轻 | New API / One API（功能更全，**仅"已在内网运行则复用"**，不为本项目专门新搭） |
| 鉴权/登录 | **后端会话（Cookie/JWT）+ 自建用户表 · 账号分配制**（F16），**认证层可插拔** | 内网应用、非全员自助；管理员开通账号（账号/岗位/角色）；认证抽象成接口，后续换公司账户零侵入 | 接公司 SSO/LDAP（后续，见 PRD 待确认 9.1-4） |
| 部署 | **Docker Compose（内网单机/小集群）** | MVP 规模够用、运维简单、便于交付到办公网服务器 | K8s（规模上来再说） |

> 选型主线：**贴合 AI 生态（Python） + 减少中间件（PG 一库多用） + 网关用轻量开源（LiteLLM Proxy，避免过重平台） + OpenAI 兼容（本地优先、可切换）**。

## 三、模型网关层设计（F15 核心）

### 3.1 方案：轻量网关（LiteLLM Proxy 首选，避免过重平台）

F15 的"自定义 Key + 多供应商（DeepSeek/GLM/GPT/中转站）+ 只配不改码"用**轻量方案**即可，无需引入带计费/多租户的重平台：

- **首选 LiteLLM Proxy**：单容器、YAML 配置多供应商，OpenAI 兼容，自带 fallback/限流/日志。新增供应商=改 YAML，应用零改动。
- **更轻的备选——应用内适配层**：供应商极少时，直接在应用配置里存 `{label, base_url, api_key, model, allow_sensitive}` 几行，按 OpenAI 协议直连，连独立网关进程都省了。
- **已有则复用——New API / One API**：若内网已经在跑，可直接复用其控制台/渠道管理，不必另搭 LiteLLM；不为本项目专门新搭。

```
后端 FastAPI ──(OpenAI 协议)──▶ LiteLLM Proxy ──┬─▶ 本地 DeepSeek (vLLM/Ollama)  [allow_sensitive]
   └─(或应用内适配层直连)                          ├─▶ DeepSeek 官方 API           [external]
                                                ├─▶ GLM (智谱)                  [external]
                                                ├─▶ GPT (OpenAI)                [external]
                                                └─▶ 中转站 (relay)              [external]
```

### 3.2 应用侧"模型策略"薄层（敏感路由 + 审计）

网关负责"怎么调供应商"，应用只保留一层**策略**决定"这次该用哪个分组/模型"：

```
choose_model(scene, sensitive) ->
   1. sensitive=True  -> 强制使用「本地 DeepSeek 分组」(sensitive-ok)，绝不发外部
   2. sensitive=False -> 用配置的默认模型（演示可指向互联网模型）
   3. 调网关 /v1/chat/completions（stream=True 时 SSE 透传前端）
   4. 记录调用审计：scene、是否敏感、用的模型、tokens、耗时、结果
```

- **敏感隔离**：LiteLLM 里给敏感场景配独立的 `model_group`/别名，只指向本地 DeepSeek；应用对敏感合同固定走该别名，从机制上保证不出网（呼应 PRD 6.2）。
- **应用配置项**（`app_model_config`，少量即可）：`default_model`、`sensitive_model`（固定本地）、`gateway_base_url`、`gateway_token(加密)`。Key 不明文回显。
- **演示切换**：把 `default_model` 指向网关里配好的互联网模型渠道即可，无需改码（满足"演示用互联网模型"）。

### 3.3 降级与容错
- 限流/失败重试、渠道健康检查、按权重负载均衡：**优先用 LiteLLM Proxy 自带能力**（YAML 配 fallback/重试）。
- 本地 DeepSeek 不可用时：**敏感合同**不自动切外部，提示"稍后重试/排队"；**非敏感**可按网关渠道策略降级。
- 长合同：应用侧分条款调用（见 4.1），避免单请求过大。

## 四、AI 方案

### 4.0 合同类型识别与类型化抽取（F1/F13/F17）

不同合同类型关注字段、规则、适用法条都不同，需**按类型走不同抽取逻辑**。设计成"识别 + 可注册的类型策略"，MVP 只支持采购合同，但为多类型扩展（F13）预留。

**类型注册表（策略模式）**：每个合同类型注册一套
```
ContractType {
  code: "purchase",            // 采购合同
  name: "采购合同",
  detect: 关键词/标题/特征规则（+模型辅助）,
  field_schema: [合同总价, 账期, 交货期, 违约金比例, 保修期, ...],  // 类型专属抽取字段
  rule_set: 该类型适用的规则,
  legal_tags: 该类型相关法条标签,
  supported: true              // MVP 仅 purchase=true
}
```

**识别与分支（上传解析后立即执行）**：
1. 对解析出的正文跑**类型识别**（先规则/关键词：如"采购合同""货物清单""乙方交付"等特征；命中弱时用模型兜底分类）。
2. 分支：
   - **识别为采购合同** → 正常进入流程，界面标注「已识别为：采购合同」（允许用户确认/纠正）。
   - **识别为其它已知类型 / 置信度低 / 无法识别** → **拦截**，提示「暂无法识别合同类型 / 暂不支持该类型」，并展示**当前支持的合同类型列表**（目前仅：采购合同）。
   - 提供**手动兜底**：用户可显式选择"按采购合同处理"以防误判卡死。
3. 识别结果决定后续用哪套 `field_schema / rule_set / legal_tags`。

**扩展方式**：新增合同类型 = 注册一个 `ContractType`（识别特征 + 抽取字段 + 规则 + 法条标签），**主流程不改**（直接支撑 F13）。

> 验收呼应：上传非采购合同 → 提示不支持 + 列出支持类型；上传采购合同 → 识别为采购合同并继续（见 PRD F17）。

### 4.1 初审编排（F2/F4/F6）

策略：**「结构化抽取 → 分条款审查 → 聚合」**的轻 Agent 流水线，而非一次性把整篇丢给模型（长合同会超窗、且定位不准）。

1. **关键字段抽取**：先用模型/规则抽取关键字段（合同总价、账期、交货期、违约金比例、保修期等），用于风险摘要与"利于/不利我方"判断。
2. **分条款审查**：对每条（优先变更条款 + 高风险条款）构造 Prompt：
   - 输入：条款原文 + 立场角色 + 命中的规则（P1）+ 按标签筛选注入的相关法条。
   - 输出（结构化 JSON）：风险等级（高/中/低）、风险说明、修改建议、法律依据引用、立场倾向。
3. **聚合**：汇总为风险摘要 + 逐条卡片，回填条款定位。

**三角色立场（F4）**：以 system prompt 注入立场（默认甲方=采购方/强势；乙方=防守；中立=公平），同一条款不同立场召回不同风险与建议。MVP 整篇按同一立场重算。

**输出契约（示例）**
```json
{
  "clause_id": "5-1",
  "title": "第五条 1 违约金比例",
  "risk_level": "mid",
  "stance": "buyer",
  "finding": "逾期违约金由 3‰ 降至 2‰，对我方采购方约束减弱",
  "suggestion": "建议维持 3‰ 或不低于公司采购违约金下限",
  "legal_basis": [{"law": "民法典", "article": "第584条", "point": "违约损失赔偿"}],
  "locator": {"page": 3, "anchor": "para_41"}
}
```

### 4.2 法律条文库（F6/F7）——不做向量 RAG，直接注入

> 决策（v1 修订）：**MVP 不做向量 RAG**。三部法律里与采购合同真正相关的条文是小集合（买卖合同/合同编通则等，至多百级），数据量小到可以**直接放进上下文**，没必要引 embedding + 向量库这套重机制。

- **范围**：民法典（采购相关：买卖合同、合同编通则等）、数据安全法、个人信息保护法，**静态导入**为结构化条目（`legal_article`：法名/编/章/条号/条文/标签）。
- **使用方式**：审查某条款时，按**条款类型 / 关键词标签**做轻量筛选（如"违约金"→召回民法典违约责任相关条文），把命中的法条**原文直接拼进 prompt**，模型在给定法条范围内择用并标注出处。
- **防幻觉**：法律依据**只能来自注入的库内条文**（引用必须可回链到 `legal_article` 条目），模型不得"杜撰法条号"；未命中相关法条则不强行给。
- **演进**：若后续扩展多合同类型（F13）使法条库显著变大，再切换为 pgvector 语义召回（`legal_article` 表结构已为此预留，迁移成本低）。

### 4.3 规则库（F8/F9，P1）
- 结构化风险规则（条件 → 风险等级/建议模板），来源：吴志刚已沉淀规则（F10，优先）或网上通用规则自定义。
- 审查时规则与模型互补：规则做确定性硬校验（如违约金比例阈值、账期上限），模型做语义/非标条款识别。

### 4.4 成本与性能

- **本地 DeepSeek**：无 token 费用，主要成本是 GPU 算力与时延 → 优化点：只审"变更条款 + 关键条款"而非全文逐条；条款级并发调用；结果缓存（同合同同版本同立场复用）。
- **外部 API（测试/演示）**：记录 token 数用于成本评估；长合同分块。

**上下文预算（按偏大估算，留足余量）**

| 组成 | 估算 tokens |
|---|---|
| 单条款原文 | 0.3k ~ 1k |
| 注入的相关法条（直接拼接，按大算：~10 条，每条 ~200 tokens） | ~2k |
| 命中规则 + 立场/输出格式 system prompt | ~1k |
| 合计（单条款审查） | **约 4k ~ 5k tokens** |

- 即便"按大一点算"，单次条款审查也就 ~5k tokens，主流模型（含本地 DeepSeek 常见 32k/64k 窗口、互联网模型）都绰绰有余，不需要为法条注入额外做截断。
- 若改为"整篇一次性审查"则会膨胀到数万 tokens（采购合同全文 + 全部法条），故坚持**分条款审查**，既控上下文又利于精确定位。
- **演示策略**：实际 demo 由演示方使用**互联网模型**（外部 API）跑，演示数据用**采购合同模板 / mock 数据（非敏感）**，因此 demo **不依赖本地 DeepSeek 是否就绪**；生产再切回本地 DeepSeek（数据不出网）。

## 五、文档/版本比对（篡改检测，F3）

> 概念：法务说的"模板"= 我方那份**标准合同文件**（比对基准），不是模板引擎。比对本质 = **两份文档对齐 diff**。**基准**可为：①我方标准合同文件；②该合同上一版本（v1→v2）。MVP 直接拿"另一份上传文件/上一版本"作基准，不依赖"模板固化"（固化是 F11/P1 增强）。

**前置：同一份合同的判定**（决定能否自动做版本比对）：
- **A 上传时手动归属（必做）**：上传时选"新合同"或"某合同的新版本"，归属确定、零误判。
- **B 编号 + C 相似度（初审后弹窗）**：初审本就要解析合同，**复用解析结果**，结束后调 `POST /api/contracts/match` 按 **合同编号 + 内容相似度（甲乙方/标的/条款标题正文）** 检测疑似同一份，弹窗提醒用户确认关联——不额外解析、不阻塞主流程。判定口径见 PRD 9.2-5。

合同比对不能用纯文本 diff（条款顺序/编号变化会乱），分三步：

1. **结构化解析**：把**基准文件**与**待审文件**都解析为条款树（依据"一、二…"「1、2…」「（1）（2）」层级 + 标题），每个叶子节点含编号、标题、正文、原文定位。
2. **条款对齐**：按"编号 + 标题 + 正文相似度（RapidFuzz）"做匹配，识别：
   - **修改**：对齐成功但正文有差异 → 字符级 diff（红删/蓝增）。
   - **新增**：待审有、基准无。
   - **删除**：基准有、待审无。
3. **关键字段变更摘要**：在 diff 基础上,对金额/账期/违约金/交货期等关键字段单独抽取前后值，生成摘要卡（对应 UI 风险摘要 4 格）。

> 采购合同样例已确认有 8 大条款（货物清单价格 / 质量技术标准 / 交货验收 / 付款 / 违约 / 不可抗力 / 争议解决 / 合同效力变更），对齐规则以此层级为基线。

## 八、数据安全与部署

### 8.1 数据安全落地（呼应 PRD 6.2）
- **生产默认本地 DeepSeek**，推理在内网 GPU 机完成，合同数据不出网。
- **敏感隔离（靠网关别名/分组）**：在 LiteLLM 里给敏感场景配只指向本地 DeepSeek 的 model 别名；敏感合同强制 `sensitive_model`（本地），从机制上不可能发往外部。
- **外部 API 仅限非敏感测试/经审批**，由网关渠道开关控制。
- 网关令牌加密存储、不明文回显；模型调用全量审计（谁/何时/哪个模型/是否敏感）。
- 全栈部署在公司办公网内，前端、后端、DB、网关、推理服务均不暴露公网。
- **合同存储安全（合规，见 PRD 待确认 9.1-5）**：
  - **传输**：全链路 TLS（内网亦启用）。
  - **静态加密**：磁盘/卷加密 或 DB 透明加密（TDE）+ 文件存储加密（MinIO SSE）；密钥用 KMS/环境变量，不入库、不硬编码。
  - **访问控制**：账号分配制 + 严格 RBAC 最小权限（按岗位/角色限制可见可改，呼应权限粒度 9.1-2）。
  - **审计（F19，独立菜单）**：关键操作全量留痕（谁/何时/做了什么/对哪份），落 `audit_log` 表，**只读追加、不可篡改**，独立菜单供管理员查阅，与模型调用审计互补。
  - **按需增强**：字段级加密（极敏感字段）、下载水印/防外传、备份加密、留存与销毁期限策略。
  - 合规依据：数据安全法 / 个人信息保护法——落"最小权限 + 加密 + 留痕 + 留存期限"。

### 8.2 部署形态
- Docker Compose 编排：`web(前端静态)` + `api(FastAPI)` + `postgres` + `minio` + **`litellm(模型网关，可选)`** +（外部已存在的）`本地 DeepSeek 推理服务`。
- 网关很轻（单容器）；若用应用内适配层可连这个容器都省去。推理服务通常独立在 GPU 机，网关通过内网地址访问其 OpenAI 兼容端口。

```
[办公网服务器]
  docker compose: web, api, postgres, minio, litellm(网关,可选)
        │ 内网调用                          │
        └────────────── litellm ───────────┴──▶ [GPU 机] vLLM/Ollama 跑 DeepSeek (:8000/v1)
                              └──▶ (非敏感) 外部 GLM/GPT/中转站
```

### 8.3 登录、岗位与立场（F16 / F14 / F4）

> 两个独立概念：**岗位角色**（销售/法务/商务/财务，F16，"你是谁"）与**立场角色**（甲/乙/中立，F4，"以什么立场解读"）。

1. **登录**（`/api/auth/login`）→ 建立会话；前端拉 `/api/me` 取用户与偏好。
2. **选岗位**（F16）：若 `remember_post=false`，进入后**弹出岗位选择**（销售/法务/商务/财务，默认法务），可勾选"记住" → `PUT /api/me/pref` 存 `default_post + remember_post=true`；若已记住则跳过、直接进入。
3. **按岗位定位**（F14）：进入合同后，按"岗位→关注**标签**"匹配定位/高亮，左侧导航点击即跳转，免翻整篇。
   - **标签机制**：解析时给每条款打 `type_tags`（价格/付款/违约金/交付/质保/保密/IP/效力/解除/争议…），定位按标签匹配而非条款号，**跨合同通用**。
   - **生效优先级**：个人 `focus_user_config` > 管理员 `focus_post_config` > **AI 默认**。
   - **AI 默认**：某岗位无 `focus_post_config` 时，由 LLM 按合同类型推荐默认关注标签（`source=ai_default`），管理员可采纳/改写。
   - **配置 UI**：管理员在「系统配置→岗位关注点」配岗位级；用户在「个人→我的关注点」做个人增减并可恢复默认。
   - MVP：内置静态默认映射兜底 + 一键定位；动态 LLM 推荐与配置 UI 列 P1（标签结构已预留）。
4. **立场切换**（F4）：甲/乙/中立在工作台顶栏切换，默认甲方；只改 AI 解读视角，不改合同内容。
